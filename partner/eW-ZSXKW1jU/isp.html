Security

Information Security Policy

Version: 1.0  ·  Effective: April 19, 2026  ·  Next review: April 2027

1. Purpose and Objectives

This Information Security Policy (ISP) establishes MintWarden's commitment to protecting the confidentiality, integrity, and availability of its information assets — including user financial data, internal systems, and infrastructure. The objectives of this policy are to:

  • Protect consumer financial data entrusted to MintWarden
  • Comply with applicable laws, regulations, and contractual obligations including partner security requirements
  • Establish accountability for information security across the organization
  • Provide a framework for identifying, assessing, and managing security risks

2. Scope

This policy applies to:

  • All MintWarden employees, contractors, and third-party vendors
  • All systems, applications, and infrastructure that store, process, or transmit MintWarden or user data
  • All data types: user PII, financial account data, access tokens, internal credentials, and audit logs

4. Core Security Controls

Access Control

  • Principle of least privilege applied to all systems
  • Role-based access control (RBAC) enforced across all internal tools
  • Access reviewed quarterly; access revoked within 24 hours of employee departure

Authentication

  • Multi-factor authentication (MFA) required on all internal systems and administrative interfaces
  • Secure tokens used for all API authentication — no plain credentials in transit
  • Consumer-facing application enforces MFA at login

Data Protection

  • All data encrypted in transit (TLS 1.2+) and at rest
  • Financial access tokens stored using strong symmetric encryption with key derivation
  • No credentials or secrets stored in source code; secrets managed via environment configuration

Vulnerability Management

  • Automated dependency scanning on all source code repositories
  • Vulnerabilities patched per defined severity SLA (Critical: 24h, High: 7d, Medium: 30d)
  • End-of-life software actively monitored and replaced per EOL Management Policy

Incident Response

  • Security incidents reported immediately to the engineering lead
  • Incident log maintained with: date, description, impact, root cause, remediation
  • Users notified of data breaches within 72 hours as required by applicable law

Logging and Monitoring

  • Access to production systems logged with timestamps and user identities
  • Logs retained for 2 years
  • Access patterns reviewed during quarterly audits

5. Third-Party Security

Third-party services are evaluated for security posture prior to integration. Access granted to third parties is limited to what is strictly necessary for the contracted purpose. We contractually require service providers to maintain security standards consistent with this policy.

6. Policy Maintenance

This policy is reviewed annually and updated when significant changes to technology or operations occur, new regulatory requirements take effect, or a security incident reveals a gap in controls.
